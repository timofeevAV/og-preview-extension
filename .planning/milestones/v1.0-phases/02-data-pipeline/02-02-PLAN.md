2. Define a `fetchWithTimeout` helper function (outside defineBackground, as a module-level utility):
   ```
   async function fetchWithTimeout(url: string, timeoutMs = 8000): Promise<Response>
   ```
   - Create an AbortController
   - Set a timeout via setTimeout that calls `controller.abort()`
   - Call `fetch(url, { signal: controller.signal, headers: { 'Accept': 'text/html' } })`
   - In the finally block, call `clearTimeout(timeoutId)`
   - Return the response

3. Inside `defineBackground(() => { ... })`, register message handlers SYNCHRONOUSLY at the top level (no awaits before registration):

   a. Handler for `getOgData` (remote URL fetch+parse):
      - Extract `url` from `message.data`
      - Validate URL: wrap `new URL(url)` in try/catch, return null if invalid
      - Check cache: `const cached = await getCachedOgData(url)` -- if cached, return cached
      - Fetch with timeout: `const response = await fetchWithTimeout(url, 8000)`
      - Validate content-type: `response.headers.get('content-type')` must include `text/html` or `application/xhtml`. If not, return null.
      - Parse: `const html = await response.text()`, then `const rawTags = parseOgTags(html)`, then `const ogData = normalizeOgData(rawTags)`
      - Cache if meaningful: if `ogData.title || ogData.image`, call `setCachedOgData(url, ogData)`
      - Return ogData
      - Wrap entire body in try/catch, log errors with `console.error('OG fetch failed:', url, error)`, return null on any error

   b. Handler for `getPageOgData` (relay to content script):
      - Extract `tabId` from `message.data`
      - This handler needs to send a message to the content script on the specified tab to extract DOM OG data
      - Use `sendMessage('getPageOgData', { tabId }, tabId)` -- check the @webext-core/messaging API for how to target a specific tab. The library typically supports a `tabId` option on sendMessage.
      - ALTERNATIVE if @webext-core/messaging doesn't support background-to-content messaging cleanly: Use `chrome.tabs.sendMessage(tabId, { type: 'getPageOgData' })` as a fallback. The content script would need to also listen via `chrome.runtime.onMessage`. Choose whichever approach compiles cleanly.
      - Return the OgData from the content script response, or null on error

4. Update `wxt.config.ts` to add `"storage"` to the permissions array if not already present. Check current config -- it already has `['activeTab', 'storage']`, so storage should be covered. Verify and leave unchanged if correct.
