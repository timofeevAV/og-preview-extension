<objective>
Fix the chrome.storage.sync MAX_WRITE_OPERATIONS_PER_MINUTE quota error caused by the hoverDelay slider in SettingsPage.tsx calling saveSettings() on every drag tick.

Purpose: Each slider drag from 0→2000 currently fires up to 40 chrome.storage.sync.set() calls. The quota is 120 writes/minute (2/second). A couple of fast drags exhaust it, causing the error. The fix reduces writes from 40-per-drag to 1-per-drag by using Radix UI's built-in onValueCommit event.

Output: SettingsPage.tsx with a localHoverDelay local state variable that drives the slider's value prop and the "X ms" display label for instant visual feedback during drag, while onValueCommit fires the single storage write only when the user releases the slider thumb.
</objective>

   ```tsx
   const [localHoverDelay, setLocalHoverDelay] = useState<number>(DEFAULT_SETTINGS.hoverDelay);
   ```

3. Change the existing `useEffect` so it syncs both `settings` and `localHoverDelay` from storage on mount. Replace:
   ```tsx
   useEffect(() => {
     getSettings().then(setSettings);
   }, []);
   ```
   with:
   ```tsx
   useEffect(() => {
     getSettings().then((s) => {
       setSettings(s);
       setLocalHoverDelay(s.hoverDelay);
     });
   }, []);
   ```

4. In the Slider JSX, make three changes:
   a. Change `value={[settings.hoverDelay]}` to `value={[localHoverDelay]}`
   b. Change `onValueChange={([v]) => update('hoverDelay', v)}` to `onValueChange={([v]) => setLocalHoverDelay(v)}`
   c. Add `onValueCommit={([v]) => update('hoverDelay', v)}` as a new prop on the Slider

5. Change the display label from `{settings.hoverDelay} ms` to `{localHoverDelay} ms`. The span currently reads:
   ```tsx
   {settings.hoverDelay} ms
   ```
   Change to:
   ```tsx
   {localHoverDelay} ms
   ```

DO NOT change anything else. The `update()` function, the other settings controls (Switch, Select for defaultTab, Select for theme), the layout, the classes — all remain identical.

DO NOT add debounce logic or useRef timers. The onValueCommit approach requires no timer.

Note on onValueCommit keyboard quirk (Radix issue #2169): when the user presses an arrow key, onValueCommit fires before onValueChange. This means a keyboard-triggered commit may be one step behind the displayed value. This is an acceptable minor UX tradeoff for a delay setting. Do not add workarounds for it.
